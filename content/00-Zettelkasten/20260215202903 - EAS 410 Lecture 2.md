- In the interests of increasing microprocessor speed, there are several techniques built into them,
	- Pipelining - All pipes in the pipeline occur simultaneously
	- Branch Prediction - predict which branch of instructions are most likely to occur next and preemptively schedule them
	- Superscalar Execution 
	- Data Flow Analysis
	- Speculative Execution
- We want to adjust the architecture and organisation of different components to balance for the performance mismatch. ie. DRAM and how we adjust interconnect widths and or include caches.
- Several improvements in chip organisation and architecture have been seen through increasing the hardware speed of the processor with smaller, more tightly packed gates, increasing the size and speed of the caches, including parallelism.
- There are several problems that exist with clock speed and logic density as higher performance typically comes at the expense of higher power draw, smaller and tighter also corresponds to higher RC product which slows down the speed that data travels.
- Multicore provides the possibility that we can increase performance without increasing the clock rate
- Amdahl's Law Deals with the potential speedup of a program using multiple processors compared to a single processor.
- The problem Amdahl's Law describes is that Software must be adapted to a highly parallel execution environment to exploit the power of parallel processing.
- Little's law describes a queuing system we can generalise to either a single server or multiple servers and allows us to find the average number of items in the queuing system $$\text{Average number of items} = \text{Rate items arrive} \times \text{Time item spends in system}$$
- Little's law is powerful for its broad application and the fact that it requires very little assumption. 
- **Instruction execution rate** is a better measure than the constant frequency $f$ that a CPU is driven by. We use the measure of average cycles per instruction (CPI) $$CPI=\frac{\sum_{i=1}^{n}(CPI_i\times I_i)}{I_c}$$
- $I_c$ is the number of machine instructions executed for the program until it runs to completion or for some defined time interval.
- $T$ processor time need needed to execute a given program $$T=I_c\times CPI\times \tau$$
- We can redefine the CPI in terms of number of cycles in the processor and in the memory $$T=I_c \times [p + (m \times k)] \times \tau$$
- $p$ is the number of processor cycles we need to **decode and execute** the instruction
- $m$ is the number of memory references needed
- $k$ is the ratio between memory cycle time and processor cycle time.
- We want to calculate the mean to benchmark systems and there are 3 common formulas for this

- Use the *AM* if the sum of all the measurements is a meaningful value, it is good for measuring execution time performance. 
- For a time based variable, *AM* has the important quality that it is directly proportional to the total time.
- *GM* gives equal weight to all the values in a dataset, any change in any of the data points regardless of magnitude will produce a significant effect on the *GM*. What this effectively does is that it reduces the effect of outliers.
- *GM* is appealing because it provides consistent results when measuring the relative performance of machines, it also used for **SPEC**.
- *HM* is used for comparing the execution rates between different machines. It is inversely proportional to the sum of the execution times
- Benchmark programes have a desireable set of characteristics that it should have: Portability through development in a high level programming language, It should be representative of a specific kind of programming domain or paradigm, It can be easily measured, It should have wide distribution.
- System Performance Evaluation Corporation (SPEC) - these guys collect the best of the best benchmarking suites aimed at evaluating computing systems
