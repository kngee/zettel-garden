- Fully associative cache organisation, any block can go into any line, it implements a search functionality and it tries to find a hit.
- Set associative mapping, it is a trade off approach. Doesn't overdo the number of circuits, it also gives you some of the benefits of fully associative mapping. 
	- Cache contains a number of sets, each set contains a number of lines and a block can map to any line the given set
	- Say we have 2 lines per set, a given block can be in one of 2 lines in only 1 set.
	- This is pretty much modelling on direct and fully associative, it is a psuedo approach.
	- We look at the tag and use set associative search to see if it is in one of our caches and then you use the set number to see where to look.
		![[Pasted image 20260227114635.png||center]]
	- If we miss it, then we assume that it is in the main memory.
	- If you have k direct mapped caches then you can do a simultaneous search so this is an advantage.
	- We are applying associative logic to be able to search through our cache in more than direct mapping registers is effectively what this approach is doing.
	- There are a number of formulas in the slides for the set associative summary.
	- ***Taking either $k$ or $v$ to 1, you will get direct mapping and fully associative mapping respectively***
	- There are larger diminishing returns to using larger and larger set associative caches. 
- Replacement algorithms answer the question of what happens when the cache has been filled. 
	- Some of these are the $LRU$ algorithm, $FIFO$ algorithm or the $LFU$ algorithm.
	- LRU is the most effective. $LFU$ is a bit more complicated in that you need to keep track of how frequently referenced a block is. ![[Pasted image 20260227115209.png|center]] and this particular image is representative of $FIFO$.
- The write policy deals with the cases that happen when a block is removed from the cache. We need to make sure that the block is written back into the main memory. This is to happen if a block has been updated while it was in the cache. 
	- There are 2 problems, what happens when one device may have access to main memory. Also, this happens when multiple processors are attached to the same bus and each processor hsa its own local cache.
	- This is to ensure that multiple access to the same memory does not create ambiguous data.
- Write through is the simplest technique, so this updates main memory when the cache is updated, it creates a lot of memory traffic and creates bottleneck.
- Write back minimizes memory writes and updates are made only in the cache. You pretty much put a lock on the block of memory so that other I/O modules can only be allowed through the cache.
- Write miss alternatives is what we do when a miss occurs at the cache level. *Write allocate*, we fetch the target from main memory, puts inot cache and processor then performs the write cycle. *No write allocate* means that the block containing the word is modified in the main memory and not loaded into the CPU.
![[Pasted image 20260227120440.png|center]]
- Either can be used with write through or write back. No write allocate is most commonly used with write through, write allocate is most commonly used with write back. You want to avoid performing two operations at once.
- Cache coherancy, we need to ensure coherance between caches that are on the same level between different devices that each have their own cache and access to the same main memory. 
- There are three possible approaches, it is important to understand that even through the write through policy is used the other caches may contain invalid data.
- There is a software approach through bus watching with write through, which is effectively like a monitoring system. A hardware approach that uses its own controller and non-cachable memory which pretty much just splits up the problem.
- As the block size increases, the hit ratio would increase because of locality. **But there are two effects that come into play by increasing the block size. Big blocks reduce number of lines and the distance between blocks increases and so there are definitely adverse effects to consider** ![[Pasted image 20260227121139.png|center]]
- For multileveled cache, there is a relationship between how large you decide to make either $L1$ or $L2$ cache.
- Unified vs Split caches, a unified cache has a higher hit rate. But the split cache eliminates contention between different units and is import for pipelining. 
- Inclusion policy, there is a upper level of where data is to be found in that level and levels below. This is good for cache coherance. Exclusion policy reverses this and defines a lower level, this approach doesn't waste cache capacity. Noninclusive policy tries neither.
